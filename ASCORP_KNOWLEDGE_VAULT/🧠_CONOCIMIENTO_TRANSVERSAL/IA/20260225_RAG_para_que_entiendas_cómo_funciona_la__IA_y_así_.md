*   **Algoritmos/Procesos (Inferencia de Ingeniería Inversa):**
    *   **Fase de Indexación de Conocimiento:**
        1.  **Recolección de Datos:** Rastreo web, scraping de contenido relevante (artículos, blogs, foros, reseñas), extracción de datos de APIs de SEO (palabras clave, volúmenes, competencia) y APIs GEO (negocios locales, puntos de interés, datos demográficos).
        2.  **Chunking:** División de los documentos y datos recolectados en fragmentos manejables (chunks) para la búsqueda. Estrategias de chunking que preserven el contexto semántico (ej., por párrafos, por secciones con solapamiento).
        3.  **Vectorización (Embeddings):** Cada chunk se convierte en un vector numérico de alta dimensión utilizando un modelo de embedding. Estos vectores capturan el significado semántico del chunk.
        4.  **Almacenamiento en Vector DB:** Los vectores (junto con metadatos y una referencia al texto original) se almacenan en una base de datos vectorial para una búsqueda eficiente.
    *   **Fase de Recuperación (Retrieval):**
        1.  **Vectorización de la Consulta:** La consulta del usuario (ej., "mejores restaurantes italianos en Palermo Buenos Aires" o "ideas de contenido para blog sobre IA en marketing") se transforma en un vector de embedding.
        2.  **Búsqueda Semántica:** Se realiza una búsqueda de k-vecinos más cercanos (k-NN) en la base de datos vectorial para encontrar los chunks más relevantes semánticamente a la consulta. Algoritmos como HNSW, IVFFlat.
    *   **Fase de Aumento (Augmentation):**
        1.  **Construcción del Prompt Contextual:** Los chunks recuperados se concatenan y se insertan en un prompt dirigido al LLM, junto con la consulta original del usuario y una instrucción clara para que el LLM base su respuesta *exclusivamente* en la información proporcionada.
        2.  **Ingeniería de Prompt:** Se formulan instrucciones detalladas para guiar al LLM a generar el tipo de respuesta deseado (ej., "Genera un artículo de blog sobre X usando la siguiente información...", "Responde a la reseña Y basándote en estos datos del negocio...").
    *   **Fase de Generación (Generation):**
        1.  **Respuesta del LLM:** El LLM procesa el prompt aumentado y genera una respuesta coherente, contextualizada y basada en la información recuperada, minimizando las "alucinaciones".
    *   **Aplicación a SEO:** Generación de contenido para blogs, landing pages, respuestas a preguntas frecuentes, meta descripciones, esquemas de datos estructurados, análisis de intención de búsqueda y auditorías de contenido.
    *   **Aplicación a GEO:** Optimización de perfiles de Google My Business, creación de contenido específico para ubicaciones, generación de respuestas personalizadas a reseñas locales, identificación de palabras clave geolocalizadas, análisis de competencia local.

## 📝 BITÁCORA DE DETALLES "INVISIBLE"
*   **La Calidad de los Embeddings es Crucial:** La elección de un modelo de embedding adecuado para el dominio de datos (ej., embeddings entrenados en datos de marketing/SEO/GEO) puede mejorar drásticamente la relevancia de la recuperación. Los embeddings son el "idioma" de la búsqueda semántica.
*   **Estrategias de Chunking Dinámicas:** No todos los documentos se benefician del mismo tamaño de chunk. Para SEO/GEO, la granularidad debe ser tal que el contexto local o temático no se diluya ni se sobrecargue, a veces usando superposiciones para mantener la continuidad.
*   **Filtraje de Metadata:** Más allá de la búsqueda por similitud vectorial, el filtrado de metadatos (ej., filtrar documentos por fecha, autor, categoría de negocio, ubicación geográfica) es esencial para la precisión de RAG en casos de uso reales.
*   **Relevancia vs. Diversidad:** Balancear la recuperación de los documentos más *relevantes* con la *diversidad* de fuentes o perspectivas puede enriquecer la respuesta del LLM. Algoritmos como Maximum Marginal Relevance (MMR) pueden ayudar.
*   **Actualización y Sincronización de Datos:** Para SEO y GEO, la información es volátil. Establecer pipelines automáticos para actualizar la base de datos vectorial (re-embedding y re-indexing) con los datos más recientes de la web o APIs es un "hack" fundamental para mantener la relevancia.
*   **Prompt Engineering de Doble Capa:** Además del prompt para RAG, puede haber un prompt "superior" que orqueste la llamada al RAG y refine la salida final del LLM para ajustarse al tono de marca o formato de publicación deseado.
*   **Evaluación Métrica de RAG:** Es vital no solo evaluar la calidad de la generación del LLM, sino también la precisión y la exhaustividad de la fase de recuperación. Métricas como Recall@k y Mean Reciprocal Rank (MRR) son clave.
*   **Costo de Operación:** Si bien potente, RAG introduce costos adicionales por el almacenamiento en bases de datos vectoriales, el cómputo de embeddings y las llamadas más complejas al LLM. Optimizar el pipeline es crucial.
